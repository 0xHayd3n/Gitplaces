How should the grouping data be structured?

All three approaches produce identical UI — the difference is in the code and data model.

A

Two fields on each language

Add domainCategory and ecosystemCategory to every language entry in languages.ts. Toggle switches which field drives the sidebar grouping.

Pros

  • Fully explicit — audit any language at a glance
  • Easy to edit one language independently
  • No indirection

Cons

  • ~140 language entries each need a second field assigned
B

Mapping table

Keep the existing category field (becomes "ecosystem"). Add a DOMAIN_MAP object that maps each ecosystem category → domain. No changes to individual language entries.

Pros

  • Minimal data change — don't touch ~140 entries
  • Ecosystem grouping already exists

Cons

  • Indirect — must cross-reference two places to see a language's domain
  • All languages in an ecosystem category share one domain (may be too coarse)
C

Central groupings config

Define a GROUPINGS config object outside the language entries. Each grouping is a record mapping a group name → list of language keys. Languages themselves stay unchanged.

Pros

  • Adding a third grouping later is trivial
  • No changes to language entries at all

Cons

  • Most indirection — groupings and languages are fully decoupled
  • Easy to forget a language when adding a new grouping

My recommendation: A — two fields. Explicit and auditable, and the one-time cost of assigning both categories to ~140 languages is low.
